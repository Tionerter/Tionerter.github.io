# <font color=#f05b72 >行为树</font>

行为树是由根节点和行为节点组成的树状数据结构.其中的行为节点是NPC可以执行的单个操作,每个行为节点都可能具有子行为节点,从而使算法体现树状结构.

每个行为节点都定义了一个前提条件(该条件指定了代理人执行此行为的条件),以及一个动作,该动作指定了代理人在执行此行为时实际执行的操作.该算法从树的根节点开始,先检查当前行为节点的前提条件,然后再依次检查每个子行为节点.在树的每个层级上,只能选择一个行为节点.因此,如果选择执行了一个行为,则不会检查其任何同级节点,但会继续检查其子级.相反,如果某项行为节点的前提条件未返回true,则该算法会跳过检查该行为节点的任何子节点,而是转至下一个同级节点.一旦抵达树的末端,算法就可以决定要运行的行为的优先级,然后根据优先级依次执行每个行为.

执行行为树的算法如下：

> * Make root node the current node
>
> * While current node exists,
>
>   * Run current node’s precondition
>   * If precondition returns true,
>     * --Add node to execute list
>     * --Make node’s child current node
>
>   * Else,
>     * --Make node’s sibling current node
>
> * Run all behaviors on the execute list

行为树的优势在于其简单直接,因此可以快速实现.而且树是无状态的,因此算法无需记住先前正在运行的行为即可确定应在给定帧上执行的行为.此外,行为可以（并且应该）写成彼此完全不知道,因此从角色的行为树中添加或删除行为不会影响树的其余部分的运行,这缓解了FSM常见的问题,即每个状态必须知道其他每个状态的转换条件.

同时行为树的可扩展性也是其优势.很容易从基础算法开始,开始添加额外的功能.常见的添加是行为on_start / on_finish函数,这些函数在行为首次开始以及行为完成时运行.也可以实现不同的行为选择器.例如,一个父行为可以规定,与其选择一个子行为来运行,不如轮流运行每个子行为,或者随机选择一个子行为来运行.实际上,如果需要,可以基于utility system类型的选择器来运行子行为.前提条件也可以编写为响应事件而触发,从而使树可以灵活地响应代理.另一个流行的扩展是将单个行为指定为非排他性的,这意味着如果运行了某个行为节点的前提条件,但行为树可以在该级别上继续检查其同级对象.

行为树虽然简单而强大,但并不总是选择AI算法的最佳选择.由于每次选择行为时,树都必须从根开始运行,因此运行时间通常大于有限状态机的运行时间.此外,根据目标平台的不同,实现代码可能包含大量条件语句,这些条件语句可能非常慢.另一方面,在处理能力受限制的情况下,评估树中的所有可能行为可能会很慢.行为树和FSM都是AI算法的有效实现,所以程序员需要去决定什么是最好的.

因为行为本身是无状态的,因此在创建行为时还需要格外小心.例如,假设有一个NPC逃避战斗.一旦远离该地区,“逃跑”行为可能会停止执行,而接管的最高优先级行为可能会使该NPC重新返回战斗区域,从而使该NPC不断在两种行为之间循环.
